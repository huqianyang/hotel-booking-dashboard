### 已确定选题
- 项目名称：酒店预订数据可视化分析与预测系统
- 英文名：Hotel Booking Data Visualization Analysis & Prediction System
- 简称：HBDVAP System
- 数据集：Kaggle "Hotel Booking Demand"
- 数据规模：119,390 条记录、32 个字段，满足课程要求的“不少于 50000 条、至少 8 个字段”。
- 核心机器学习任务：基于 `is_canceled` 字段进行酒店预订取消二分类预测。
- 扩展功能方向：实时统计、多维度分析、趋势分析、客户画像和预测结果可视化。

### 数据集初步探索结果

- 模型评估指标页面展示应使用中文主名，并可在括号中保留英文缩写，便于老师对应机器学习概念，例如：准确率（Accuracy）、精确率（Precision）、召回率（Recall）、F1 值（F1-score）、AUC 值。

### 原文与黑板图二次精读补充
- 老师原文明确把课程设计拆成三层：机器学习模拟训练模块、外部 Web 应用模块、框架整合运维模块；外部应用再拆前端和后端，因此三人协作最自然的颗粒度是“机器学习一人、后端/数据库/框架一人、前端/可视化一人”。
- 前端 UI 示例不是炫酷大屏，而是 Flask/PyECharts 课程 Web 应用：顶部黑色导航，菜单为“首页、数据查询、数据可视化、评估与预测”；内容区以浅色页面、搜索框、选项卡、单张大图为主。
- “首页”颗粒度：不是复杂操作页，核心是让老师知道系统干什么，并展示核心数据或系统简介。
- “数据查询”颗粒度：必须体现 MySQL 数据库基本操作，至少支持查询；老师原文也提到可以扩展添加、修改等关系型数据库操作。该页是 Web 应用是否完整的关键证据。
- “数据可视化”颗粒度：要分历史数据可视化和实时数据可视化；历史可视化需覆盖时间、分类、文本、数值型数据，多种图表必须服务预测/分析目标。
- “地图交互”颗粒度：老师强调不是下拉框筛选优先，而是图表之间直接交互；地图点击某国家/地区后联动展示该地区相关柱状图或指标，这是必做项。

- 黑板图补充了数据库与框架边界：MySQL 存历史/结构化表；Redis/Ready 是本项目必须纳入的实时缓存层，用于保存实时流处理中间状态、最新指标、最近预测记录或实时图表增量数据；Storm 处理流式 tuple；外部应用/前端通过 Flask 接口或 JSON 读取处理结果。
- 用户已确认老师明确要求 Redis，因此 Redis 不再作为仅报告说明的扩展项；后续文档和设计改为 MySQL + Redis 双数据库：MySQL 负责历史数据、查询、模型指标和持久化结果，Redis 负责实时缓存、最新指标、最近预测和实时增量可视化数据。
- 关于“Redis → Storm 实时处理”的课堂口径：技术上可以成立，尤其在课程项目中可把 Redis 用作模拟实时数据缓冲区/消息队列，数据生成器按时间顺序写入 Redis List 或 Redis Stream，Storm 通过 Redis Spout 读取并计算实时指标；但在标准大数据架构中更常见的是 Kafka → Storm → Redis。若老师强调 Redis + Storm，可以采用“Redis 作为实时数据缓冲输入 + Storm 处理 + Redis/MySQL 输出”的简化课程链路，并在报告中说明 Kafka/Flume 是更标准的消息采集与缓冲层。
- Flume 和 Kafka 的处理口径：若老师要求同时体现 Flume/Kafka/Storm/Redis，则采用完整实时链路 `模拟日志文件 → Flume → Kafka → Storm → Redis + MySQL`。其中 Flume 负责采集模拟实时日志，Kafka 负责消息缓冲和削峰，Storm 负责实时计算和预测调用，Redis 负责最新实时指标与最近预测缓存，MySQL 负责历史数据和结果持久化。Redis 不放在 Storm 上游时更符合标准链路；若按黑板图强调 Redis + Storm，则 Redis 可作为 Storm 下游实时缓存，并在前端实时接口中体现其作用。
- 模拟实时数据发送速率应按老师原文口径设计：检查历史数据的时间属性和时间粒度，例如按小时、天或月份统计单位时间内的数据量，再按压缩比例换算成演示时每秒发送几条数据。老师原文明确提到“以小时为单位”“看这个小时的数据量多少”“算一下一秒钟应该发送几个数据”，本质是在用定时器模拟实时数据发送流程。
- 机器学习颗粒度必须从“模型文件”提升到“训练过程文档化”：特征、标签、训练/测试划分、模型选择、模型对比、训练集/测试集效果、评估指标、超参数、损失/收敛、过拟合/欠拟合都要能在页面或报告中找到对应说明。

### realtime_metrics 表用途核对
- 当前项目目录未发现 `.py`、`.sql`、`.java` 实现文件；`realtime_metrics` 主要存在于 `任务文档/需求分析说明书.md` 与 `任务文档/系统规划设计说明书.md`。
- 文档定位：`realtime_metrics` 保存 Java Storm 生成的实时聚合指标快照，用于 MySQL 持久化和课程报告复现；Redis 负责最新实时展示缓存，不承担长期历史保存。
- 设计字段：`metric_id`、`metric_name`、`metric_value`、`metric_type`、`window_start`、`window_end`、`updated_at`。
- 使用链路：Java Storm 消费 Kafka 事件后调用 Flask 预测 API，预测结果写入 `prediction_results`，聚合指标写入 `realtime_metrics`，最新摘要/趋势/最近预测等同步写入 Redis。
- 页面依赖：首页的实时摘要/链路状态、数据可视化页的实时风险监控模块、实时指标 API 或最近预测 API 可读取 Redis + MySQL；课程验收也要求 MySQL 中存在实时指标数据。
- 必要性判断：如果只做当前运行中的实时看板，可用 Redis 替代；如果要满足“Storm 指标落库”“系统重启后可复现”“报告截图/验收查询 MySQL”，则建议保留。
