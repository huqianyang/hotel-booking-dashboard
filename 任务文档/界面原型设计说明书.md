# 酒店预订数据可视化分析与预测系统界面原型设计说明书

## 一、设计目标

本说明书用于说明“酒店预订数据可视化分析与预测系统”的前端页面原型设计。系统面向课程展示、答辩演示和酒店预订取消风险分析场景，页面设计围绕“数据查询、可视化分析、取消风险预测、实时流处理展示”四个核心目标展开。

界面原型设计遵循以下目标：

1. **贴近真实业务系统**：页面不直接堆叠课程检查项，而是以酒店经营风险分析、订单查询、实时监控和模型辅助决策为业务主线。
2. **覆盖课程功能要求**：通过首页、数据查询、数据可视化、评估与预测四个页面覆盖 Flask Web 应用、MySQL 查询、PyECharts/ECharts 可视化、机器学习预测、Flume/Kafka/Storm/Redis 实时链路等要求。
3. **便于后续实现**：页面布局尽量采用 Flask + HTML + Tailwind/jQuery + PyECharts/ECharts 可实现的结构，避免过度复杂的大屏动画。
4. **便于答辩解释**：每个页面都有明确职责，能够说明数据从哪里来、图表如何联动、实时数据如何刷新、模型预测结果如何产生。

## 二、整体界面风格

系统采用“课程 Web 应用增强版”风格，而不是纯炫酷大屏风格。整体视觉以浅色页面、卡片式内容区、顶部导航、清晰图表区域为主，既接近老师示例中的 Flask/PyECharts 项目形态，又通过统一配色、指标卡和图表联动增强展示效果。

整体风格特点：

- 顶部使用统一导航栏，包含首页、数据查询、数据可视化、评估与预测四个主入口。
- 页面主体采用浅色背景和卡片分区，便于展示表格、图表和模型指标。
- 图表优先服务业务问题，不在页面上直接标注“时间数据可视化、关系数据可视化”等课程标签。
- 课程要求对应关系放在说明书和报告中说明，页面本身保持真实业务系统观感。
- 实时数据通过局部刷新体现，不做整页刷新，避免页面闪烁和交互中断。

## 三、页面导航结构

系统规划为四个主页面：

| 页面 | 页面职责 | 主要数据来源 |
|---|---|---|
| 首页 | 展示系统总览、实时指标、风险态势、演示环境状态 | Redis 实时缓存 + MySQL 汇总数据 |
| 数据查询页 | 展示 MySQL 历史订单查询、筛选、详情、编辑和删除操作 | MySQL `hotel_bookings` |
| 数据可视化页 | 展示围绕取消风险的多维图表分析、图表联动和实时风险监控 | MySQL 历史统计 + Redis 实时指标 |
| 评估与预测页 | 展示单条订单预测、模型评估详情和批量预测记录 | MySQL `prediction_results`、`model_metrics`、`realtime_metrics` + Redis 最新预测 |

页面之间的关系如下：

```text
首页
├── 查看系统整体运行状态
├── 进入数据查询页查看具体订单
├── 进入数据可视化页分析取消风险来源
└── 进入评估与预测页查看模型预测和评估结果
```

## 四、首页原型设计

首页作为系统入口，主要回答“系统当前运行情况如何、实时订单处理到哪里、取消风险是否异常”三个问题。

页面主要模块：

1. **系统概览区**
   - 展示系统名称、项目用途和当前模拟日期。
   - 说明系统围绕酒店预订取消风险进行数据分析和预测。

2. **实时核心指标区**
   - 累计预订量。
   - 今日处理订单数。
   - 高风险订单数。
   - 平均取消概率。
   - 当前取消率。

3. **今日风险趋势区**
   - 按模拟业务时间展示订单处理量和取消风险变化。
   - 时间段采用业务模拟时间，例如 08:00-10:00、10:00-12:00、12:00-14:00。

4. **高风险订单区**
   - 展示最新高风险订单列表。
   - 包含订单编号、国家、渠道、客户类型、取消概率和风险等级。

5. **演示环境控制区**
   - 区分“启动演示环境”和“开始订单模拟”。
   - 启动演示环境用于启动 Flask、MySQL、Redis、Flume、Kafka、Storm 等服务。
   - 开始订单模拟用于按历史时间顺序发送模拟订单事件。

6. **服务状态区**
   - 展示 Flask、MySQL、Redis、Flume、Kafka、Storm 等组件运行状态。
   - 用于说明大数据实时链路是否正常。

首页对应原型文件：`任务文档/完整首页UI示例.html`。

## 五、数据查询页原型设计

数据查询页主要体现 MySQL 历史数据查询和关系型数据库操作能力。页面不只是静态表格，而是提供筛选、查看详情、编辑、删除和操作日志等功能。

页面主要模块：

1. **查询筛选区**
   - 支持按酒店类型、国家、渠道、客户类型、是否取消、预订时间等条件筛选。
   - 筛选条件提交后调用 Flask 查询接口，从 MySQL 中获取结果。

2. **订单结果表格**
   - 展示订单编号、酒店类型、国家、渠道、客户类型、提前预订天数、ADR、是否取消等字段。
   - 表格支持分页，避免一次性展示过多数据。

3. **订单详情区**
   - 用户点击某条订单后展示详细字段。
   - 用于解释单条订单的业务含义，也可作为后续预测页面选择订单的入口。

4. **编辑与删除入口**
   - 支持对部分演示字段进行编辑。
   - 删除采用逻辑删除或演示删除方式，避免破坏原始数据集。

5. **操作日志区**
   - 展示用户查询、编辑、删除等操作记录。
   - 该日志用于说明 Web 应用层的数据库操作行为，不等同于 MySQL 底层 binlog。

数据查询页对应原型文件：`任务文档/数据查询页UI示例.html`。

## 六、数据可视化页原型设计

数据可视化页围绕“酒店预订取消风险分析”组织图表，不直接把课程中的五类可视化名称贴在页面上。页面看起来应像真实的经营分析系统，课程要求则通过说明书和报告进行映射。

页面主要模块：

1. **经营风险总览**
   - 当前筛选订单数。
   - 当前取消率。
   - 高风险订单数。
   - 平均取消概率。

2. **预订趋势分析**
   - 展示月度预订量和月度取消率。
   - 用于观察取消风险随时间变化的趋势。

3. **取消结构分析**
   - 展示已取消和未取消订单占比。
   - 支持在不同筛选条件下查看取消构成变化。

4. **取消原因分析**
   - 展示提前预订天数、ADR 房价、押金类型、特殊需求数量等因素与取消风险的关系。
   - 用于解释哪些业务因素更容易导致订单取消。

5. **客户与渠道分析**
   - 展示 Online TA、Offline TA、Direct、Groups 等渠道取消风险。
   - 支持结合客户类型分析不同客群的取消表现。

6. **地区风险分布**
   - 使用国家地图展示不同来源国家或地区的取消率。
   - 地图颜色深浅表示取消风险高低。

7. **高风险特征标签**
   - 以词云或标签云形式展示高风险特征，例如长提前期、无押金、团队订单、高 ADR、特殊需求少等。
   - 用业务化语言表达文本数据可视化，不在页面中写“文本数据可视化”。

8. **当前筛选订单摘要表**
   - 展示当前筛选条件下的订单摘要。
   - 用户点击图表后，该表格同步更新，帮助从图表跳转到具体订单。

### 6.1 图表联动规则

数据可视化页采用统一筛选状态管理。用户点击图表中的国家、月份、渠道、客户类型或风险因素后，页面将更新全局筛选条件，并重新加载相关图表和订单摘要表。

示例联动规则：

- 点击地图中的某个国家：趋势图、渠道排行、取消原因图和订单摘要表同步切换到该国家数据。
- 点击趋势图中的某个月份：地图、渠道分析、风险标签和订单摘要表同步切换到该月份数据。
- 点击渠道排行中的某个渠道：趋势图、地区分布、风险标签和订单摘要表同步切换到该渠道数据。
- 点击高风险标签：筛选具备该风险特征的订单，并更新相关图表。

### 6.2 课程可视化要求对应关系

页面中不直接显示课程标签，但在设计上自然覆盖课本五类可视化要求：

| 页面业务模块 | 对应课程可视化类型 | 说明 |
|---|---|---|
| 预订趋势分析 | 时间数据可视化 | 使用月份或模拟业务时间展示预订量与取消率变化 |
| 取消原因分析 | 关系数据可视化 | 展示业务因素与取消风险之间的关系 |
| 取消结构分析 | 比例数据可视化 | 展示已取消和未取消订单占比 |
| 高风险特征标签 | 文本数据可视化 | 用标签云展示高风险业务特征 |
| 地区风险分布 + 图表联动 | 复杂数据可视化 | 地图结合多维筛选和订单表格形成复杂联动分析 |

数据可视化页推荐以 `任务文档/数据可视化页UI示例-联动版.html` 作为定稿基础。

## 七、评估与预测页原型设计

评估与预测页已定稿为“三页签模型工作台版”，页面重点是让用户完成单条订单预测、查看模型评估证据、查看批量预测处理结果。

页面分为三个顶层页签：

### 7.1 订单预测

该页签用于对单条订单进行取消风险预测。

主要内容：

- 订单选择区：从 `hotel_bookings` 历史订单或实时订单中选择待预测订单。
- 订单特征摘要：展示酒店类型、国家、渠道、客户类型、提前预订天数、ADR 等关键字段。
- 预测结果区：展示取消概率、风险等级、预测标签。
- 风险原因解释：展示影响该订单风险的关键因素。
- 建议动作：给出保留订单、重点跟进、提醒确认等业务建议。

### 7.2 模型评估详情

该页签用于展示模型训练和测试过程，满足机器学习评估要求。

主要内容：

- 最终采用模型说明。
- 模型选择原因。
- 训练集 / 测试集划分说明。
- 逻辑回归与随机森林模型对比。
- 准确率、精确率、召回率、F1 值等指标。
- 混淆矩阵。
- 过拟合 / 欠拟合判断。

当前设计中删除 ROC 曲线和过度复杂指标，避免页面过于教学化和答辩解释压力过大。

### 7.3 批量预测记录

该页签展示 Storm 或后端在实时订单处理中按时间窗口聚合得到的批量预测结果。

主要内容：

- 当前模拟日期。
- 时间段，例如 08:00-10:00、10:00-12:00、12:00-14:00。
- 批次编号。
- 处理订单数。
- 高风险订单数。
- 平均取消概率。
- 最新批次排在最上方。

批量预测记录不保留“手动导入”来源，避免页面上出现没有入口支撑的功能。

评估与预测页对应原型文件：`任务文档/评估与预测页UI示例-模型工作台版.html`。

## 八、页面交互说明

系统页面交互遵循“从总览到明细、从图表到订单、从预测到解释”的路径。

主要交互包括：

1. 首页点击高风险订单，可跳转到评估与预测页查看单条预测详情。
2. 数据查询页点击订单，可查看订单详情，也可作为后续预测输入。
3. 数据可视化页点击图表元素，可联动更新其他图表和底部订单摘要表。
4. 数据可视化页筛选结果可与数据查询页保持字段口径一致。
5. 评估与预测页可从订单选择进入预测结果，再查看模型评估或批量预测记录。

## 九、实时刷新机制

系统采用分级自动刷新机制，体现流处理系统的实时性，同时避免过于频繁请求导致页面和后端压力过大。

| 模块 | 刷新频率 | 说明 |
|---|---:|---|
| 首页核心指标 | 3 秒 | 展示订单处理量、高风险订单数、平均取消概率等快速变化指标 |
| 趋势图追加点 | 5 秒 | 从 Redis 获取最新趋势点，追加到当前趋势图 |
| 批量预测记录 | 5 秒 | 刷新最新实时处理窗口的批量预测结果 |
| 服务状态 | 10 秒 | 检测 Flask、MySQL、Redis、Flume、Kafka、Storm 等组件状态 |

实时数据链路为：

```text
Python 实时数据生成器
→ booking_events.log
→ Flume
→ Kafka
→ Java Storm
→ Flask 预测 API
→ Redis + MySQL
→ Flask 实时接口
→ 前端局部刷新
```

数据读取策略：

- 页面首次加载和历史查询优先读取 MySQL。
- 最新实时指标、趋势增量、最近预测记录优先读取 Redis。
- MySQL 保存预测结果和实时指标快照，用于系统关闭后的历史查询和课程报告复现。
- 前端只更新局部模块，不刷新整个页面。
