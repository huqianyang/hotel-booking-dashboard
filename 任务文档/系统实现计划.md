# 系统实现计划

## 一、目标

本计划用于指导“酒店预订数据可视化分析与预测系统”的实际开发。当前阶段只确定开发顺序、模块边界和阶段交付，不提前展开完整接口文档和字段规范。字段规范应在数据清洗和 MySQL 建表阶段固化，接口文档应在各页面 API 实现阶段细化。

## 二、总体实现原则

1. 先做单机闭环，再接大数据链路。
2. 先做历史数据，再做实时模拟数据。
3. 先让页面能展示真实数据，再做图表联动和自动刷新。
4. 先让 Flask 能读写 MySQL 和 Redis，再让 Storm 写入 Redis 和 MySQL。
5. 每个阶段都要能单独运行、截图和解释。

## 三、推荐开发顺序

```text
阶段 1：项目骨架与运行环境
阶段 2：数据清洗与字段中文映射
阶段 3：MySQL 建表与历史数据导入
阶段 4：机器学习训练与模型评估
阶段 5：Flask 基础应用与页面模板
阶段 6：数据查询页
阶段 7：数据可视化页
阶段 8：评估与预测页
阶段 9：Redis 实时缓存与实时接口

阶段 11：首页实时联动与整体联调
阶段 12：测试、截图和报告材料整理
```

## 五、阶段 1：项目骨架与运行环境

目标：先搭出可以启动的 Flask Web 应用。

主要工作：

- 创建 `requirements.txt`。
- 创建 `app.py` 和 `config.py`。
- 创建 `backend/routes/page_routes.py`。
- 创建基础模板 `templates/base.html` 和首页 `templates/index.html`。
- 创建统一样式 `static/css/main.css`。

验收标准：

- 执行 `python app.py` 后 Flask 能启动。
- 浏览器能打开首页。
- 顶部导航能看到：首页、数据查询、数据可视化、评估与预测。

## 六、阶段 2：数据清洗与字段中文映射

目标：把 Kaggle 原始数据清洗成系统可直接使用的数据。

主要工作：

- 将原始 CSV 放入 `data/raw/hotel_bookings.csv`。
- 编写 `scripts/clean_data.py`。
- 输出 `data/processed/hotel_bookings_cleaned.csv`。
- 处理缺失值，例如 `children`、`country`、`agent`、`company`。
- 构造到店日期字段。
- 增加中文展示字段，例如酒店类型、客户类型、市场渠道、是否取消。

验收标准：

- 清洗后数据量仍大于 50000。
- 核心字段存在：`is_canceled`、`hotel`、`lead_time`、`adr`、`country`、`arrival_date`。
- 页面展示用中文字段，模型训练和数据库底层可保留英文变量名。

## 七、阶段 3：MySQL 建表与历史数据导入

目标：完成关系型数据库基础。

核心表：

1. `hotel_bookings`：清洗后的历史订单。
2. `prediction_results`：单条和批量预测结果。
3. `model_metrics`：模型训练与评估指标。
4. `realtime_metrics`：实时指标快照。

主要工作：

- 编写 `scripts/init_mysql.py` 创建数据库和表。
- 编写 `scripts/import_mysql.py` 导入清洗后的历史数据。
- 编写 `backend/db.py` 封装 MySQL 连接。

验收标准：

- MySQL 中存在四张核心表。
- `hotel_bookings` 表记录数大于 50000。
- Flask 后端可以连接 MySQL。

## 八、阶段 4：机器学习训练与模型评估

目标：训练酒店预订取消预测模型，并保存模型与评估指标。

主要工作：

- 编写 `scripts/train_model.py`。
- 使用逻辑回归和随机森林进行模型对比。
- 保存最终模型到 `models/cancellation_model.pkl`。
- 保存特征列到 `models/feature_columns.json`。
- 保存评估指标到 `models/metrics.json`。
- 将模型评估结果写入 MySQL 的 `model_metrics` 表。

评估指标：

- 准确率
- 精确率
- 召回率
- F1 值
- 混淆矩阵
- 训练集得分
- 测试集得分
- 最终采用模型

验收标准：

- 能训练出可加载的模型文件。
- 能输入一条订单特征并返回取消概率和风险等级。
- 评估与预测页有可展示的模型评估数据。

## 十二、阶段 8：评估与预测页实现

目标：实现三页签模型工作台。

三个页签：

1. 订单预测。
2. 模型评估详情。
3. 批量预测记录。

主要工作：

- 编写 `backend/services/prediction_service.py`。
- 编写 `backend/routes/prediction_routes.py`。
- 单条预测调用模型并写入 `prediction_results`。
- 模型评估详情读取 `model_metrics`。
- 批量预测记录读取 `prediction_results` 和 `realtime_metrics`。

验收标准：

- 能选择订单并预测取消概率。
- 能展示风险等级和原因解释。
- 能展示模型评估指标。
- 批量预测记录按模拟业务时间倒序排列。

## 十三、阶段 9：Redis 实时缓存与实时接口

目标：让页面可以局部自动刷新实时数据。

主要工作：

- 编写 `backend/redis_client.py`。
- 编写 `backend/services/realtime_service.py`。
- 编写 `backend/routes/realtime_routes.py`。
- 编写 `scripts/seed_redis.py`，用于没有 Storm 时模拟 Redis 实时数据。

建议 Redis Key：

```text
realtime:summary
realtime:trend
realtime:recent_predictions
realtime:country_risk
realtime:service_status
```

刷新频率：

```text
首页核心指标：3 秒
趋势图追加点：5 秒
批量预测记录：5 秒
服务状态：10 秒
```

验收标准：

- 不启动 Storm 时，也可以通过 `seed_redis.py` 看到页面实时刷新。
- 有 Redis 数据时优先读 Redis。
- Redis 无数据时可以回退到 MySQL 最新快照。

## 十四、阶段 10：Flume / Kafka / Java Storm 实时链路

目标：打通课程要求的大数据实时处理链路。

链路：

```text
Python 实时数据生成器
→ booking_events.log
→ Flume
→ Kafka
→ Java Storm
→ Flask 预测 API
→ Redis + MySQL
```

主要工作：

- 编写 `scripts/generate_realtime_events.py`。
- 编写 Flume 配置。
- 编写 Docker Compose。
- 编写 Java Storm Topology。
- Storm 消费 Kafka 事件。
- Storm 调 Flask 预测 API。
- Storm 写 Redis 最新指标。
- Storm 写 MySQL 持久化结果。

验收标准：

- Kafka 能收到模拟事件。
- Storm 能消费事件。
- Flask 预测 API 被调用。
- Redis 中有最新指标。
- MySQL 中有预测结果或实时指标快照。

## 十五、阶段 11：首页实时联动与整体联调

- 服务状态接服务检测接口。


## 十七、开发优先级

如果时间充足，按十二个阶段完整推进。

如果时间紧，优先级如下：

1. 必做：数据清洗、MySQL 导入、Flask 页面、数据查询、基础可视化、模型训练、单条预测。
2. 次必做：Redis 实时缓存、首页实时刷新、批量预测记录。
3. 最后做：Flume / Kafka / Java Storm 真实链路。

如果大数据环境调试受阻，可以先用 `seed_redis.py` 模拟实时刷新，保证页面演示效果，再继续排查 Flume / Kafka / Storm。

## 二十一、分工注意事项

1. 成员 A、B、C 不是完全割裂开发，每个人都要理解整体流程。
2. 成员 B 负责系统整合，但不能替成员 A 完成模型训练，也不能替成员 C 完成全部页面。
3. 成员 C 可以先用假数据做页面，等 B 的接口完成后再切换到真实 API。
4. 成员 A 的字段处理会影响 B 的数据库和 C 的页面展示，因此清洗后字段要尽早稳定。
5. 如果 Flume/Kafka/Storm 调试时间不够，B 至少要保证 Redis 模拟实时刷新可演示，并保留大数据链路配置和运行说明。
6. 答辩时每个人都要能说明自己的代码位置、运行方式、输入输出和设计原因。
