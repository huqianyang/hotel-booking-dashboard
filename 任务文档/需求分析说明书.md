# 酒店预订数据可视化分析与预测系统需求分析说明书

## 一、项目概述

本项目名称为“酒店预订数据可视化分析与预测系统”。系统基于 Kaggle Hotel Booking Demand 数据集，围绕酒店预订取消问题，构建一个集数据查询、数据可视化、机器学习取消预测、实时流处理和课程报告支撑于一体的 Web 应用。

项目使用 Flask 作为后端框架，使用 PyECharts / ECharts 作为可视化展示工具，使用 Scikit-learn、Pandas、NumPy 完成数据清洗、特征工程、模型训练与模型评估。系统采用 MySQL + Redis 双数据库设计：MySQL 负责历史预订数据、数据查询、模型指标、预测结果和实时处理结果的持久化；Redis 负责实时缓存、最新指标、最近预测记录和实时增量可视化数据。

根据课程要求，系统需要真实跑通 Flume、Kafka、Storm、Redis 与 MySQL 的整合链路。实时链路采用“Python 实时数据生成器 → booking_events.log → Flume → Kafka → Java Storm Topology → Redis + MySQL → Flask API → 前端页面”的方式：Python 脚本按历史时间字段模拟预订事件持续产生，Flume 采集日志，Kafka 进行消息缓冲，Java Storm 消费事件、调用 Flask 预测 API、计算实时指标，并将预测结果和实时统计结果分别写入 MySQL 与 Redis，最终由 Flask API 提供给前端展示。

本项目不需要制作答辩 PPT，最终提交物以完整源代码、运行说明和 Word 课程设计报告为主。项目文件、文档文件和交付物文件名统一使用中文。

## 二、项目目标

- 构建一个可运行的 Flask Web 系统，展示酒店预订数据查询、可视化分析、取消预测和实时风险监控结果。
- 使用 Hotel Booking Demand 数据集完成数据清洗、字段处理、特征工程、模型训练和模型评估。
- 基于 `is_canceled` 字段构建二分类取消预测任务，并记录完整训练过程。
- 采用“逻辑回归 + 随机森林分类器”的简化模型对比方案：逻辑回归作为基础对比模型，随机森林分类器作为主模型。
- 使用 PyECharts / ECharts 展示围绕取消率预测主线的图表，并自然覆盖课本五类数据可视化：时间、关系、比例、文本、复杂数据可视化。
- 提供图表之间的交互联动，支持点击国家、月份、渠道、客户类型、风险标签等维度后联动更新指标卡、图表和订单明细。
- 使用 Docker Compose 真实运行 Flume、Kafka、Java Storm 和 Redis，并跑通实时数据处理链路。
- 使用本机 MySQL 保存历史预订数据、预测结果、实时指标快照和模型评估结果。
- 使用 Redis 保存实时最新指标、最近预测记录、地图风险缓存和实时增量序列。
- 支持三人协作开发，分工覆盖数据算法、后端与实时链路、前端与可视化。
- 输出结构清晰、符合课程设计要求的 Word 课程设计报告。

## 四、核心业务问题

酒店预订取消会影响酒店房间资源安排、收益管理和客户服务。系统需要回答以下问题：

- 酒店预订取消率整体情况如何？
- 取消率随月份、入住日期、预订提前期等时间因素如何变化？
- 不同国家、酒店类型、市场渠道、客户类型、押金类型下取消率有什么差异？
- 提前预订天数、平均每日房价、历史取消次数、特殊需求数量等特征与取消率之间有什么关系？
- 哪些客户标签、渠道标签或风险标签与取消风险更相关？
- 给定一条新预订信息，系统能否预测其取消概率和风险等级？
- 实时模拟预订流入时，系统能否持续计算高风险订单数量、平均取消概率、热门国家、渠道风险 Top 等实时指标？
- 页面能否通过图表联动帮助教师和用户理解“为什么会取消”和“哪些订单风险更高”？

## 五、功能范围

### 1. 必做功能

#### 1.1 数据导入与清洗

- 支持读取 `hotel_bookings.csv`。
- 支持处理缺失值、异常值和字段类型转换。
- 支持构造用于分析的时间字段，例如 `arrival_date`、`event_date`、`send_time`。
- 支持将清洗后的历史数据导入 MySQL 的 `hotel_bookings` 表。
- 数据量不少于 50000 条，字段不少于 8 个。
- 数据字段同时包含数值型字段和分类型字段。

#### 1.2 数据查询与 MySQL 基本操作

- 提供“数据查询”页面，体现 MySQL 基本检索能力。
- 支持按酒店类型、国家、月份、市场渠道、客户类型、是否取消等条件查询预订记录。
- 支持分页展示查询结果。
- 支持新增预订记录，用于体现关系型数据库新增操作。
- 支持修改预订记录的部分字段，用于体现关系型数据库更新操作。
- 支持删除预订记录，建议采用 `is_deleted = 1` 的逻辑删除方式，避免破坏原始数据。
- 查询、新增、修改、删除操作均通过 Flask API 访问 MySQL。

页面业务模块包括：

1. 取消趋势分析。
2. 取消因素分析。
3. 客户与渠道分析。
4. 地区交互分析。

#### 1.4 图表交互联动

- 支持点击世界地图中的国家或地区，联动展示该地区的取消率、渠道分布、客户类型分布和高风险订单明细。
- 支持点击月份或时间点，联动更新地图、渠道图、客户类型图和订单表。
- 支持点击市场渠道，联动更新取消率、国家分布、客户类型和订单明细。
- 支持点击客户类型，联动更新取消率趋势、渠道风险和订单明细。
- 支持点击词云中的风险标签，联动展示相关指标和订单明细。
- 联动筛选条件应在页面上可见，并支持清除筛选条件。

#### 1.5 机器学习取消预测

- 基于 `is_canceled` 字段构建二分类预测任务。
- 训练集 / 测试集划分推荐使用 8:2 或 7:3，并在页面或报告中说明。
- 基础对比模型使用逻辑回归，用于说明简单分类模型效果。
- 主模型使用随机森林分类器，用于提高非线性特征处理能力和展示特征重要性。
- XGBoost、LightGBM、LSTM 不作为必做模型，避免超过学生项目复杂度。
- 保存每个候选模型的评估指标到 `model_metrics` 表。
- 评估指标至少包含准确率、精确率、召回率、F1 值和 AUC。
- 页面或报告需要说明特征、标签、模型选择理由、模型对比、训练集表现、测试集表现、超参数、过拟合/欠拟合和收敛情况。
- 标记最终选中的上线模型。
- 保存最终模型文件，由 Flask 负责加载和推理。
- Java Storm 不直接加载 Python 模型文件，而是调用 Flask 预测 API。

#### 1.6 评估与预测页面

- 页面提供模型训练过程说明，包括特征、标签、训练集/测试集划分和模型选择理由。
- 页面展示逻辑回归与随机森林分类器的指标对比。
- 页面展示混淆矩阵、ROC 曲线、AUC、特征重要性和过拟合/欠拟合判断。
- 页面提供预订取消预测表单。
- 用户输入酒店类型、提前预订天数、国家、市场渠道、房价等关键特征。
- Flask 调用已训练模型返回取消概率、预测标签和风险等级。
- 预测结果写入 `prediction_results` 表。

#### 1.7 实时流处理链路

- Python 脚本逐行读取 `hotel_bookings.csv`，按 `event_date` 的历史时间顺序模拟新预订事件持续产生。
- 数据生成器根据历史时间粒度和压缩比例生成 `send_time`，用于课程演示中的实时发送。
- 模拟预订事件写入 `booking_events.log` 或 Flume 监听目录。
- Flume 采集业务事件日志并发送到 Kafka。
- Kafka 作为消息缓冲层保存预订事件。
- Java Storm Topology 消费 Kafka 中的预订事件。
- Java Storm 调用 Flask `/api/取消预测` 接口，获得取消概率、预测标签和风险等级。
- Java Storm 计算实时聚合指标，例如实时处理量、高风险订单数、平均取消概率、热门国家、渠道风险 Top 和酒店类型占比。
- Java Storm 将预测结果和实时指标快照持久化写入 MySQL。
- Java Storm 将最新指标、最近预测记录、地图风险缓存和实时增量序列写入 Redis。
- Flask API 同时读取 MySQL 和 Redis，向前端提供历史数据、实时缓存数据和增量更新数据。

#### 1.8 MySQL 与 Redis 存储

MySQL 至少包含以下数据表：

- `hotel_bookings`：保存 CSV 历史导入数据，作为数据查询、历史分析、模型训练和实时模拟事件的主数据来源。
- `prediction_results`：保存单条预订的预测结果，通过 `booking_id` 关联 `hotel_bookings`。
- `realtime_metrics`：保存 Java Storm 生成的实时聚合指标快照。
- `model_metrics`：保存逻辑回归、随机森林等候选模型的整体评估结果。

Redis 至少承担以下职责：

- 保存实时核心指标，例如当前处理量、高风险订单数、平均取消概率和更新时间。
- 保存最近若干条预测记录，用于实时风险监控页面滚动展示。
- 保存实时趋势增量序列，用于前端定时追加新点。
- 保存国家、渠道、客户类型等维度的实时风险缓存，用于地图和联动图表快速展示。
- 保存实时链路状态信息，用于展示 Flume、Kafka、Storm、Redis 的最新处理状态。

## 十、实时链路说明

课程项目中的标准实时链路如下：

```text
hotel_bookings.csv
  ↓
Python 实时数据生成器按 event_date 顺序逐行读取
  ↓
按压缩比例生成 send_time，并持续追加 booking_events.log
  ↓
Flume 监听日志文件
  ↓
Kafka 接收预订事件
  ↓
Java Storm Topology 消费 Kafka 事件
  ├── 调用 Flask 预测 API，预测结果写入 MySQL
  ├── 计算实时聚合指标，指标快照写入 MySQL
  └── 最新指标 / 最近预测 / 增量序列写入 Redis
  ↓
Flask API 查询 MySQL + Redis
  ↓
PyECharts / ECharts 数据可视化页面展示
```

真实业务中，用户提交预订后通常由业务后端直接写入 MySQL，保证交易主链路稳定；同时业务系统输出事件日志或通过 CDC/binlog 工具采集变更，再进入 Kafka/Storm 等流处理链路。本课程项目不做 MySQL binlog/CDC 采集，使用 Flume 采集模拟业务事件日志。Redis 不替代 Kafka，而是作为 Storm 下游实时缓存层，服务实时可视化和增量更新。
