# 字段规范说明书

## 一、文档目的

本文档用于统一“酒店预订数据可视化分析与预测系统”的字段口径，作为三人并行开发前的基础契约。

- 成员 A 使用本文档确定模型训练特征、标签和指标解释字段。
- 成员 B 使用本文档设计 MySQL 表结构、查询接口和预测接口。
- 成员 C 使用本文档确定页面展示列名、筛选条件、图表维度和中文文案。

当前字段规范以 `app/data/cleaning.py` 中的 `CLEANED_BOOKING_COLUMNS` 和 `app/database/schema.py` 中的 `hotel_bookings` 表结构为准。

## 二、字段命名原则

1. 数据库字段和 Python 变量名使用英文 snake_case。
2. 页面展示优先使用中文字段名。
3. 英文字段保留原始含义，中文字段用于页面、报告和答辩解释。
4. 模型训练使用英文变量名，避免中文字段影响 Scikit-learn 特征处理。
5. 所有页面接口返回字段应尽量保持与数据库字段一致，前端负责展示中文标题。

### 5.3 不建议作为模型特征的字段

| 字段名 | 原因 |
|---|---|
| booking_id | 主键，无业务预测意义 |
| hotel_name | 中文展示字段，和 `hotel` 重复 |
| is_canceled_label | 标签中文展示字段，和 `is_canceled` 重复 |
| arrival_date | 可用于时间分析，但第一版模型可先不纳入，避免时间泄漏解释复杂 |
| event_date | 用于实时模拟排序，不作为预测特征 |
| country_name | 中文展示字段，和 `country_code` 重复 |
| market_segment_name | 中文展示字段，和 `market_segment` 重复 |
| meal_name | 中文展示字段，和 `meal` 重复 |
| deposit_type_name | 中文展示字段，和 `deposit_type` 重复 |
| customer_type_name | 中文展示字段，和 `customer_type` 重复 |
| reservation_status | 该字段可能包含预订最终结果，容易造成标签泄漏 |
| reservation_status_date | 和最终状态相关，第一版模型不使用 |
| is_deleted | 系统逻辑删除字段，不属于业务预测特征 |

## 六、成员 B：数据库和后端字段边界

1. `hotel_bookings` 表必须保持 41 列字段顺序和清洗 CSV 一致。
2. 查询接口默认排除 `is_deleted = 1` 的记录。
3. 编辑接口只允许修改演示字段，不应修改模型标签 `is_canceled`。
4. 逻辑删除只更新 `is_deleted = 1`，不物理删除数据。
5. 后端接口返回字段名使用英文，前端表头使用中文。
6. 后端需要保留 `booking_id` 作为所有详情、预测和编辑操作的主键。

## 八、字段变更规则

1. 任何人新增、删除、重命名字段前，必须先同步修改本文档。
2. 字段变更必须同时检查：清洗函数、MySQL schema、导入脚本、测试、接口契约、页面展示。
3. 模型特征字段变更必须由成员 A 确认，并同步更新 `feature_columns.json`。
4. 页面展示字段变更必须由成员 C 确认，避免接口字段存在但页面无处展示。
5. 数据库字段变更必须由成员 B 确认，避免导入脚本和 MySQL 表结构不一致。
